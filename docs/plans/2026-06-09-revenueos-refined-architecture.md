## 1. Schema Changes

### Enums Removed → Dynamic Text

| Before (pgEnum) | After (plain text) | Why |
|---|---|---|
| `seniorityEnum` | `text` — agent decides | Agent discovers titles that don't fit rigid bands. Let it store any value. |
| `taskTypeEnum` | `text` — agent decides | Task types emerge from signal analysis. Hardcoding limits agent creativity. |
| `signalTypeEnum` (already removed) | `text` — dynamic from ICP | Already agreed. Signal types derive from ICP matching. |

### Enums Kept (System-Level)

| Enum | Why Kept |
|---|---|
| `orgStatusEnum` | System lifecycle: `onboarding → active → paused → churned` |
| `runStatusEnum` | System execution: `pending → running → completed → failed` |
| `taskStatusEnum` | System tracking: `pending → in_progress → completed → skipped → failed` |

### JSONB Properties Pattern

Contacts and organisations get a flexible `properties` column:

```ts
// On contacts table
properties: jsonb("properties").$type<Array<{
  key: string;       // e.g. "email", "phone", "department", "location"
  value: string;
  type: "text" | "number" | "url" | "date";
}>>().default([]),
```

```ts
// On organisations table
properties: jsonb("properties").$type<Array<{
  key: string;       // e.g. "employee_count", "funding_stage", "industry"
  value: string;
  type: "text" | "number" | "url" | "date";
}>>().default([]),
```

**Why:** The agent discovers properties we can't predict. LinkedIn gives location, department, tenure. Exa gives employee count, funding stage, industry vertical. Rather than adding columns, the agent adds key-value pairs dynamically.

### New Fields on Organisations

```ts
// The "Soul.md" — generated by initialisation agent from uploaded PDFs + domain
organisationMd: text("organisation_md"),

// References to S3 files used to build the ICP
icpSourceFiles: jsonb("icp_source_files").$type<Array<{
  url: string;
  filename: string;
  uploadedAt: string;
  extractedText?: string;  // extracted PDF text (stored for re-processing)
}>>().default([]),
```

## 2. Initialisation — SEEDED (not built for demo)

**Decision:** The initialisation flow (PDF upload → extract → run agent → generate Organisation.md) is **seeded via script**, not built as a live feature. The core demo is the autonomous research agent running on stage — onboarding is a setup step.

### What We Seed
- 5 demo orgs (Grab hero + Canva, Sea Group, Tokopedia, Gojek)
- Each org has a **hand-crafted `organisationMd`** (Soul.md) with full ICP profile
- All orgs pre-set to `status: "active"` with `honchoPeerId` and initial scores

### S3 Role
S3 is kept in the architecture for **per-workspace skill markdown storage** — not PDF uploads. Future: users can save custom market skills as markdown files in S3.

### Future (post-hackathon)
- Build Phase 0 properly: PDF upload, pdfjs-dist extraction, initialisation agent
- Build onboarding page with file upload form

## 3. Exa API — Deep Dive Tool Design

### Exa Endpoints We Use

| Endpoint | Tool | Purpose |
|---|---|---|
| `POST /search` | `exa_people_search` | Find people at target companies |
| `POST /search` | `exa_company_deep_dive` | 8-angle parallel company intelligence |
| `POST /search` | `exa_person_deep_dive` | 6-angle parallel stakeholder profiling |
| `POST /contents` | (used within tools) | Get full page content for URLs |
| `POST /answer` | `exa_answer` | Direct Q&A with citations |

### Deep Dive Search Strategy

Both company and person deep dives run **multiple parallel searches** across different Exa categories:

#### `exa_company_deep_dive` — 8 parallel queries
1. Funding & investment → category: `news`
2. Leadership changes → category: `news`
3. Expansion & growth → category: `news`
4. Product & strategy → category: `news`
5. Financial reports → category: `financial report`
6. Regulatory & compliance → category: `news`
7. ICP-specific signals → category: `company`
8. Competitive landscape → category: `news`

#### `exa_person_deep_dive` — 6 parallel queries
1. Speeches & talks → category: `news`
2. Podcast appearances → category: `news`
3. Conference & event attendance → category: `news`
4. LinkedIn activity → category: `linkedin profile`
5. Social interests & hobbies → category: `news`
6. Recent news mentions (priorities) → category: `news`

### Source Attribution (MANDATORY)
Every claim — company or person level — must include:
1. The **exact source URL**
2. The **specific paragraph or quote** confirming the claim

## 4. Honcho as Persistent System Prompt Memory

### Change: Recall → Inject into System Prompt

**Before:** Agent calls `honcho_recall` as a tool during the run.

**After:** Before every research run, the system:
1. Calls `honcho_recall(peerId, "Summarize everything known about this account")`
2. Injects the recall result INTO `agent/instructions.md` as a section
3. The agent starts each run already knowing its accumulated context

### Implementation

```ts
// In the agent run trigger (server action or schedule)
const memory = await honchoRecall(org.honchoPeerId);
const systemPrompt = buildSystemPrompt({
  organisationMd: org.organisationMd,   // The "Soul.md"
  marketSkill: loadSkill(org.hqCountry),
  accumulatedMemory: memory,             // Honcho recall
  orgProperties: org.properties,
});
// Pass to Ash agent session
```

### System Prompt Structure

```markdown
# RevenueOS Agent

## Account Identity (Organisation.md)
{organisationMd — the backbone ICP profile generated during onboarding}

## Market Context
{loaded from skills/markets/{country}/SKILL.md}

## Accumulated Knowledge (Self-Learning Memory)
{Honcho recall — everything learned across all prior runs on this account}

## Research Instructions
{standard research workflow}
```

This means:
- **First run:** Only Organisation.md + market skill (no prior memory)
- **Subsequent runs:** Organisation.md + market skill + ALL accumulated knowledge
- The agent literally gets smarter every run without any code changes

## 7. Key Design Principles (Restated)

1. **Agent-decided types** — No rigid enums for things the agent discovers. Use `text` + `properties` JSONB.
2. **Organisation.md as Soul.md** — Every account agent has a backbone identity generated from the customer's own documents.
3. **Persistent self-learning** — Honcho recall injected into system prompt, not just a tool call. The agent starts smarter each run.
4. **Exa categories** — Use built-in categories (`linkedin profile`, `company`, `news`) for dramatically better search quality.
5. **Exa Answer** — Use `/answer` endpoint for direct Q&A with citations, not just raw search.
6. **PDFjs-dist** — No native deps, works in serverless, extracts text from uploaded pitch decks.
